# 面试深度讲解

## 一句话介绍

这是一个基于 GitOps 与 SLO 的云原生发布稳定性平台，用一个 FastAPI 微服务串联 CI/CD、Kubernetes 生产化治理、供应链安全、可观测性、渐进式发布、故障演练和运维自动化。

## 讲解主线

### 1. 为什么不是普通 CI/CD

普通项目通常只做到代码提交后自动构建镜像并部署。本项目把 CI/CD 拆成质量门禁、测试、制品构建、安全扫描、部署验证和 GitOps 发布多个阶段，每个阶段都有明确输入、输出和失败策略。

### 2. Kubernetes 如何体现生产化

重点讲：

- readinessProbe 防止异常 Pod 接流量。
- livenessProbe 处理进程不可恢复。
- HPA 应对流量变化。
- PDB 保证节点维护时最小可用。
- NetworkPolicy 实现网络最小权限。
- SecurityContext 降低容器逃逸风险。
- ResourceQuota 防止单服务耗尽命名空间资源。

### 3. 为什么使用 SLO/Error Budget

资源指标不能直接表达用户体验。SLO 把可用性和延迟量化，并用 Error Budget 作为发布速度和稳定性之间的调节器。

### 4. 为什么 burn-rate 告警优于固定错误率告警

固定错误率告警无法区分短暂尖刺和持续退化。多窗口 burn-rate 同时考虑短窗口和长窗口，可以减少误报，也能更快发现严重事故。

### 5. GitOps 与 Canary 的价值

GitOps 让集群状态可审计、可回滚；Canary 通过逐步放量降低爆炸半径；Prometheus AnalysisTemplate 把人工观察变成自动化发布准入。

### 6. 运维开发能力体现在哪里

`opsctl` 把常见排障动作、SLO 报告、故障演练、容量建议、发布证据留存和 incident report 生成固化为工具，体现从“执行运维动作”到“开发运维平台能力”的转变。

### 7. 平台治理能力体现在哪里

顶级大厂更关注平台能力是否能规模化复制。本项目用 `policies/release-gates.yaml` 表达生产准入，用 Kyverno/OPA 示例表达集群准入，用 `change-risk` 把变更映射到不同门禁，用 `prod-readiness` 把人工 checklist 变成可执行检查。这样可以把个人经验升级为团队级平台机制。

## 高频追问

### 为什么 `maxUnavailable` 设置为 0？

为了保证滚动发布期间不主动减少可用副本，尤其在副本数较少时避免容量瞬间下降。

### 自动回滚是否能解决所有发布问题？

不能。自动回滚只负责快速恢复服务，根因分析仍需要结合日志、指标、链路、CI 产物和变更记录。

### 为什么 Trivy 在 PR 和 main 上策略不同？

PR 阶段更重视快速反馈，main 分支更接近发布路径，因此对 CRITICAL 漏洞应更严格阻断。

### 为什么要生成 SBOM？

SBOM 让团队知道镜像包含哪些依赖和版本，在漏洞爆发时可以快速判断影响范围。

### 为什么要做发布证据包？

因为生产发布不只需要知道“这次发布成功了”，还要能在故障后快速回答“发布了哪个镜像、对应哪个 commit、当时副本和重启情况如何、SLO 是否退化”。`evidence-bundle` 把镜像、Deployment 状态、重启次数和 Prometheus SLO 快照固化成 JSON，便于发布审计、故障复盘和回滚决策。

### 为什么要引入变更风险分级？

不同变更的风险不同。文档变更不应该和 RBAC、NetworkPolicy、生产 values、SLO 阈值变更走完全相同的门禁。风险分级可以把发布成本和风险匹配起来：低风险快速反馈，高风险必须经过策略、安全、部署验证、release-gate 和证据归档。

### 为什么混沌实验要有停止条件？

混沌实验的目标是验证系统韧性，不是无限制破坏系统。停止条件用于控制 blast radius，例如 Ready Pod 数低于阈值、fast burn 告警触发、P95 延迟超过上限或错误率超过阈值时立即停止实验并保留现场。这样能体现 SRE 对风险控制、用户影响和实验伦理的理解。

### 为什么 HPA 要配置 scaleDown 稳定窗口？

扩容需要尽快响应流量上涨，但缩容过快容易在流量抖动、滚动发布或节点维护时造成容量不足。scaleDown 稳定窗口保留一段时间的冗余容量，用更高的资源成本换取稳定性。
